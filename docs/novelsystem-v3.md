# 小说创作辅助系统设计文档

> **核心理念**：作者主导，AI协从；冲突驱动，故事自演进  
> 版本：3.2

## 1. 项目概述

本系统是一个为小说作者设计的 **AI深度协作环境**，与普通写作软件不同，它：

- 显式追踪故事中的时间轴、角色状态、地点、全局变量。
- 为每个角色（含环境）分配独立的 **Agent**，每个Agent拥有自己的记忆、性格、语言风格。
- 故事推进由 **矛盾冲突** 驱动：系统自动检测当前状态中的潜在冲突，推荐事件，作者选择后AI自动演绎完整场景。
- **作者永远在驾驶位**：所有生成内容需作者确认，AI只提供建议和辅助写作。

---

## 2. 设计理念

- **极简沉浸**：编辑器纯粹、无干扰，支持全屏黑暗模式。
- **结构化叙事**：帮助作者理清时间线、人物关系和情节脉络。
- **数据安全第一**：本地存储为主，自动保存 + 多版本历史。
- **隐私优先**：所有AI分析可选本地模型，用户数据不离开设备。
- **可扩展**：支持自定义Agent、冲突类型、导出格式。

## 3. 核心数据模型：状态追踪模板

故事中的所有动态元素都通过状态模板管理，支持时间轴快照与版本回溯。

### 3.1 角色状态模板

```json
{
  "角色ID": "hero_01",
  "姓名": "林逸风",
  "性格描述": "重情义，冲动，不善表达",
  "语言风格": "短句，生气时吼，平时低沉",
  "动态状态": {
    "HP": 100,
    "内力/魔力": 80,
    "情绪": "平静",
    "位置": "清风镇客栈",
    "装备": ["青云剑", "布衣"],
    "物品栏": ["干粮x3", "地图", "密信"],
    "人际关系": { "师妹": 75, "反派": -60 },
    "临时效果": ["左臂轻伤（持续2天）"]
  },
  "不变属性": { "年龄": 25, "背景": "父亲失踪" }
}
```

### 3.2 地点/世界状态模板

```json
{
  "地点ID": "town_01",
  "名称": "清风镇",
  "状态": {
    "治安": 60,
    "天气": "小雨",
    "时间": "戌时",
    "关键NPC": ["老铁匠", "老板娘"],
    "事件标记": ["悬赏告示已贴"],
    "可互动物品": ["药铺灵芝"]
  }
}
```

### 3.3 全局剧情变量

```json
{
  "主线进度": 0.3,
  "关键抉择": { "救不救老人": "已救" },
  "伏笔池": ["月圆觉醒", "神秘玉佩"]
}
```

### 3.4 冲突历史（新增）

记录角色间已发生的冲突类型及结果，避免重复生成相同矛盾。

```json
{
  "冲突ID": "conf_001",
  "时间戳": "故事第12天",
  "参与角色": ["hero_01", "shimei_01"],
  "类型": "言语冲突",
  "结果": "不欢而散"
}
```

## 4. 时间轴与章节管理

- **可视化时间轴**：横向为故事时间（可自定义单位：天/小时），每个章节为一个时间块。
- **状态快照**：每个章节开始/结束时自动保存完整状态JSON，支持回滚。
- **时间推进**：每生成一个新章节，AI根据内容估算耗时，作者可手动修正。

---

## 5. 多角色Agent系统

每个主要角色拥有独立的 Agent，包含：

- 性格档案、语言风格、动态状态（从模板同步）。
- 短期记忆（最近发生的事件）。
- 长期记忆（关键事件，存储于向量库）。
- 当前目标（驱动Agent决策）。

### 5.1 Agent能力

| 能力 | 说明 |
|------|------|
| 生成对话 | 根据当前场景和对手，输出符合性格的台词。 |
| 做出决策 | 在多选项中选出最符合性格的行动。 |
| 主动建议 | 当角色状态重大变化时，静默提示作者。 |

### 5.2 Agent间通信协议

- 消息格式：`{ "text": "...", "emotion": "愤怒", "action": "瞪了一眼" }`
- **状态同步**：Agent不直接修改全局状态，而是输出 **状态变化建议**（如好感度+3），由协调器统一合并后交作者确认。
- **协调器**负责多Agent对话的 **回合制调度**。

## 6. Agent独立记忆系统

每个Agent拥有 **自己独特的记忆**，同一事件在不同Agent记忆中可能表述不同（体现视角差异）。

### 6.1 记忆类型

| 类型 | 容量 | 存储 | 用途 |
|------|------|------|------|
| 短时记忆 | 最近20~50条 | 循环队列（内存+本地文件） | 保持对话连贯 |
| 长时记忆 | 无限制 | 向量数据库（如Chroma） | 影响长远决策和情感 |

### 6.2 记忆内容示例

- 林逸风的记忆："师妹在暴雨夜骂了我，我很难过。"
- 师妹的记忆："当时我很担心他，但他什么都不说，我只好发脾气。"

### 6.3 记忆的读写权限

| 操作 | 作者 | Agent自身 | 其他Agent | 协调器 |
|------|------|-----------|-----------|--------|
| 读取自己记忆 | ✅ 全部 | ✅ 全部 | ❌ | ✅（用于检索注入） |
| 写入/修改记忆 | ✅ | ✅（仅自己） | ❌ | ❌ |
| 删除记忆 | ✅ | ✅（仅自己） | ❌ | ✅（清理过期短时记忆） |

Agent之间不能直接读写对方记忆，但可通过对话间接交换信息（例如A告诉B某件事，B可能将其存入自己的记忆）。

### 6.4 记忆影响行为

- **决策时**：检索相关长时记忆，注入Prompt，修正好感度权重。
- **生成对话时**：可能引用具体过去事件（"你还记得三年前那个雨夜吗？"）。
- **情感演化**：检索到悲伤记忆会使当前情绪偏向悲伤。

### 6.5 技术实现

- **短时记忆**：Agent对象内存数组，定期保存到本地JSON。
- **长时记忆**：Chroma或LanceDB + 本地嵌入模型（如all-MiniLM-L6-v2）。
- **检索**：每次Agent调用前做向量相似度搜索，Top-K=3~5，延迟<50ms。
- **Prompt注入模板**：见附录。

### 6.6 作者交互界面

- **记忆浏览器**：点击角色头像，查看/编辑/添加记忆。
- **记忆对比视图**：并排显示两个角色对同一事件的记忆差异。
- **快捷记忆注入**：选中文本 → 右键 → "让{角色}记住这件事"。

## 7. 环境描写Agent

独立的Agent，专门负责生成环境描写，且拥有自己的 **风格配置** 和 **独立记忆**（例如记住之前描写过的天气变化，保持连贯）。

### 7.1 职责

- 根据地点、天气、时间、冲突类型生成环境描写。
- 环境描写必须服务于冲突氛围（如雨夜烘托紧张）。
- 可被自动调用（场景开头/结尾）或手动触发。

### 7.2 风格预设

| 风格 | 特点 | 适用场景 |
|------|------|----------|
| 诗意型 | 多用比喻、通感，长句渲染 | 内心冲突、浪漫场景 |
| 冷峻型 | 短句，白描，压抑感 | 悬疑、恐怖 |
| 动作型 | 环境与角色动作结合 | 战斗、追逐 |

作者可自定义风格。

## 8. 事件驱动与冲突引擎

核心循环：

```
冲突检测 → 作者选择 → 事件编排 → 生成场景 → 状态更新 → 重复
```

### 8.1 冲突类型库

| 类型 | 参与者 | 典型状态变化 | 环境需求 |
|------|--------|-------------|----------|
| 武力冲突 | 多角色 | HP下降，位置改变 | 紧张，有障碍物 |
| 言语冲突 | 多角色 | 好感度下降，情绪愤怒 | 私密或公共场合 |
| 内心冲突 | 单角色 | 情绪波动，目标改变 | 孤独，安静 |
| 环境冲突 | 角色 vs 自然 | HP下降，被迫移动 | 恶劣天气 |
| 社会冲突 | 角色 vs 群体 | 声望变化，被通缉 | 城市，法庭 |

### 8.2 冲突检测器

- **规则检测**（实时，低开销）：
  - 好感度 < 30 且同地点 → 言语冲突候选
  - HP < 20 且敌对角色距离近 → 武力冲突候选
  - 角色目标互斥 → 争夺冲突候选
- **AI辅助**（可选，每章结束后运行）：将状态快照发给大模型，补充候选。

### 8.3 事件编排器（生成一个场景的步骤）

1. 复制当前状态快照作为起始点。
2. 调用环境Agent生成开场描写（传入冲突类型、地点、时间）。
3. 按冲突类型调度角色Agent轮流生成动作/对话：
   - 武力冲突：动作优先，每轮更新临时HP/位置。
   - 言语冲突：对话轮次，每轮更新临时好感度/情绪。
4. **实时状态模拟**：每步生成后立即更新临时状态，并传给下一个Agent。
5. **冲突解决判断**：达到停止条件（HP<0、轮数上限、作者预设）结束。
6. 调用环境Agent生成结尾描写。
7. 输出完整场景文本 + 状态更新JSON。
8. 等待作者确认或微调。

### 8.4 作者控制点

- 选择冲突（从候选列表或手动创建）
- 设置初始条件（如某角色HP=20）
- 调节Agent活跃度（攻击性/理性）
- 打断重试、手动修改生成的文本
- 最终确认状态更新

## 9. 作者交互流程示例

| 步骤 | 作者动作 | 系统/AI动作 |
|------|----------|-------------|
| 1 | 写了几章后，系统后台自动运行冲突检测 | 弹出通知："发现2个潜在冲突：①林逸风与师妹好感度低 ②反派跟踪" |
| 2 | 点击"查看冲突" | 展示候选卡片，每个卡片包含冲突类型、参与角色、可能后果。 |
| 3 | 选择"言语冲突：林逸风与师妹争吵" | 弹出配置面板：可设置起始情绪、地点、是否允许和解等。 |
| 4 | 调整参数后点击"生成场景" | 事件编排器运行：环境Agent生成雨夜客栈环境 → 师妹Agent先说话 → 林逸风Agent回应… 输出完整场景和状态变化。 |
| 5 | 阅读生成内容，修改部分对话，点击"确认" | 保存章节到时间轴，更新全局状态（好感度-12，情绪变为沮丧）。同时将此事存入两个角色的长时记忆（各自表述）。 |
| 6 | 继续写作 | 系统后台再次检测新冲突… |

## 10. 技术实现建议

### 10.1 技术栈

| 模块 | 推荐技术 |
|------|----------|
| 前端 | Electron + Vue 3 / React |
| 本地数据库 | SQLite + IndexedDB（存储状态快照） |
| 时间轴UI | HTML Canvas 或 Fabric.js |
| 多Agent协调 | Python asyncio / Node.js Worker Threads（顺序调用即可） |
| AI推理 | 云端：OpenAI API / Claude；本地：Llama 3 8B Q4 + Ollama |
| 向量记忆（长时） | Chroma 或 LanceDB（本地） |
| 规则引擎 | 自研轻量级规则（JSON条件匹配） |
| 短时记忆 | JSON 文件 + 内存 |

### 10.2 关键实现点

- **状态快照版本控制**：每章保存一个完整状态JSON，支持回滚到任意历史点。
- **Prompt模板管理**：为每种Agent（角色、环境、逻辑检查等）维护独立模板，动态注入状态和记忆。
- **冲突检测器**：每章结束后运行，先规则，后可选AI辅助。输出候选事件列表。
- **事件编排器的状态模拟**：维护临时状态对象，每步解析模型输出的状态变化（如HP-15）并立即更新。
- **隐私模式**：所有AI调用默认使用本地模型，只有用户主动开启联网才上传数据。

## 12. 可行性分析

> 总体判断：**理论可行，但完整实现的难度极大。**  
> 以下按严重程度逐一分析核心困难。

### 12.1 LLM 调用成本与延迟

这是整个系统最大的瓶颈。根据 8.3 节定义的事件编排流程，一个"言语冲突"场景的 API 调用量为：

```
环境Agent开场 → 角色A第一句 → 角色B回应 → 角色A再回应 → 角色B再回应 → 环境Agent结尾
```

假设 3 轮对话，至少需要 **6-10 次** LLM API 调用。

| 方案 | 单场景成本（估计） | 单场景延迟 | 中文小说对白质量 |
|------|-------------------|-----------|-----------------|
| Claude / GPT-4 云端 | ¥2 - ¥20 | 30-120 秒 | 优秀 |
| Llama 3 8B Q4 本地 | 免费（仅电费） | 60-300 秒 | 较差，生硬 |
| Qwen 2.5 14B 本地 | 免费 | 30-150 秒 | 中等 |

**关键矛盾**：文档 2.2 节提出的"隐私优先 + 纯本地模型"与"高质量中文创作"难以兼得。目前开源本地模型的中文虚构写作能力，尤其在仙侠/武侠类对白方面，与云端前沿模型差距显著。建议在隐私和质量之间做显式取舍，而不是假设本地模型可以达到同等水平。

### 12.2 多 Agent 协调的不稳定性

文档 8.3 节描述的"临时状态"机制是一个精巧的设计，但在实践中面临多层风险：

- **结构化输出不稳定**：LLM 被要求同时输出 `{ "text": "...", "emotion": "...", "state_changes": [...] }` 格式，偶发 JSON 格式错误（字段名拼错、括号不匹配），需要大量容错代码。
- **角色漂移**：Agent 在对话第 4-5 轮时倾向于重复自己，或逐渐偏离性格设定。这在回合数较多的场景中尤其明显。
- **级联失败**：一个 Agent 的输出异常会导致后续 Agent 接收到错误的状态上下文，进而产生更差的输出。整个场景生成过程缺乏"单点恢复"能力。
- **中文角色一致性**：让 LLM 稳定模拟"重情义、冲动、不善表达"这种微妙性格，在长篇对话中保持一致，是当前所有模型（包括云端）的短板。

### 12.3 品质与一致性

- **独立视角记忆**（文档 6.2 节）是整个设计中最有创意的部分，但让 LLM 始终以"林逸风的视角"记住事件、同时让"师妹"以不同视角记住同一件事，需要极强的角色一致性。当前模型在短对话中可以做到，但跨章节长时间维持几乎不可能。
- **向量检索精度不足**：文档 6.5 节建议使用 all-MiniLM-L6-v2 做中文嵌入。该模型对中文的语义区分能力有限 —— 输入"师妹生气了"，可能检索到"师妹在笑"的记忆。建议换用中文专用嵌入模型（如 BGE-large-zh），但会显著增加本地资源消耗。
- **冲突检测过于机械**：仅靠好感度/HP 阈值（文档 8.2 节）会产生大量低质量建议（"这两个人好感度低 → 言语冲突"反复出现）。真正的叙事张力远比数值阈值复杂。

### 12.4 工程量评估

| 阶段 | 内容 | 预估工时（单人） |
|------|------|-----------------|
| V1.0 基础编辑器 | Electron + 编辑器 + 章节管理 | 2-3 个月 |
| V1.5 状态追踪 | JSON 模板引擎 + 时间轴 UI | 2-3 个月 |
| V2.0 单 Agent | LLM 接入 + Prompt 管理 + 短时记忆 | 2-3 个月 |
| V2.5 多 Agent | 协调器 + 向量库 + 角色管理 UI | 3-4 个月 |
| V3.0 冲突引擎 | 规则引擎 + 事件编排器 + 冲突 UI | 3-4 个月 |
| V3.5 高级功能 | 对比视图 + 逻辑检查 + 导出 | 2-3 个月 |
| **合计** | | **14-20 个月** |

以上为单人全职估算。UI 侧的每一项（记忆浏览器、冲突卡片、对比视图）都对应数百到数千行前端代码。

### 12.5 隐私 vs 质量：核心取舍

文档同时追求"所有 AI 调用默认使用本地模型"和"高质量小说生成"。这两个目标在 2026 年依然存在根本性张力。建议在系统设计中显式引入一个 **"隐私-质量"滑块**：

- 滑块偏左（隐私优先）：所有调用走本地 Ollama，部分功能（如多 Agent 对话协调）降级或关闭
- 滑块偏右（质量优先）：使用云端 API，可选加密传输
- 让用户自己选择，而不是替用户做决定

## 13. 替代方案

基于可行性分析，以下四种替代方案分别针对不同痛点，可独立或组合采用。

### 方案 A：单 Agent 场景生成（推荐起步方式）

**针对**：12.1 成本延迟 + 12.2 多 Agent 不稳定性

放弃多 Agent 独立调度，改为**一次 LLM 调用生成完整场景**。将所有角色性格、当前状态、冲突类型注入同一个 Prompt，由单一模型直接写出整个场景（含对话、动作、环境描写）。

**优势**：
- API 调用量减少 80%（1 次 vs 6-10 次）
- 避免多 Agent 协调的级联失败风险
- 大模型擅长一次性写长场景，实际质量不亚于多 Agent 分步生成

**代价**：
- 失去"角色视角独立性"的体验（作者无法看到每个 Agent 的独立决策过程）
- 失去了文档 5.2 节中"Agent 不互相读记忆"的精妙设计

**建议**：V2.0 阶段优先采用方案 A，验证场景生成质量后再考虑是否演进到多 Agent 架构。

### 方案 B：Web App 替代 Electron

**针对**：12.4 工程量

用 Next.js + IndexedDB + 浏览器端向量库（LanceDB WASM）开发 Web 应用，替代 Electron 桌面应用。

**优势**：
- 开发速度提升 2-3 倍（无需处理 Electron 的窗口管理、IPC、打包等）
- 热更新体验更好
- 利用浏览器现有能力：IndexedDB（本地存储）、Web Workers（后台计算）、Canvas（时间轴）

**代价**：
- 无法直接访问本地文件系统（但可通过 File System Access API 缓解）
- 无法像 Electron 那样深度集成系统菜单和快捷键
- 后续可用 Tauri 打包为桌面应用，成本低

### 方案 C：LLM 摘要替代向量记忆

**针对**：12.3 向量检索精度

放弃向量数据库，改用**定期 LLM 摘要**来管理长时记忆：

1. 每章结束后，将短时记忆队列发给 LLM
2. LLM 将其总结为 3-5 句"角色印象"（如"林逸风最近经历了师妹的责骂，内心有愧疚，但不愿表达"）
3. 摘要直接注入到未来对话的 Prompt 中

**优势**：
- 省掉向量数据库的部署和维护
- LLM 摘要理解深层含义，比向量相似度匹配更精准
- 对小说创作而言，几十字的"角色印象"往往比 5 条原始记忆片段更有用

**代价**：
- 每章结束额外一次 LLM 调用（但量很小，成本可忽略）
- 无法精确检索"三年前某个雨夜"的具体细节（但对于创作来说，摘要反而对 LLM 更有用）

### 方案 D：冲突模板替代自动检测

**针对**：12.3 机械冲突检测

放弃基于规则的自动冲突检测（8.2 节），改为让作者从**冲突模板库**中手动选择。

模板示例：

```json
{
  "模板名": "追责型言语冲突",
  "触发条件": "A 认为 B 做错了某事",
  "适用关系": "师徒 / 情侣 / 手足",
  "典型情绪曲线": "质问 → 辩解 → 情绪升级 → 爆发或和解",
  "环境建议": "私密空间，氛围压抑"
}
```

**优势**：
- 避免低质量重复建议（好感度低 → 吵架 的循环）
- 作者掌握叙事节奏控制权，模板更像是"编排助手"而非"自动决策者"
- 模板库本身可以不断积累，成为作者的知识资产

**代价**：
- 失去了"系统自动发现作者没意识到的冲突"的惊喜感
- 但 AI 辅助检测（8.2 节的可选方案）仍然可以保留，作为模板选择的补充

### 替代方案组合建议

| 阶段 | 推荐组合 |
|------|---------|
| **快速原型**（1-2 月） | 方案 B（Web App）+ 方案 C（LLM 摘要）+ 方案 D（冲突模板） |
| **V2 功能验证** | 方案 A（单 Agent 场景生成）替代多 Agent，加入云端 API 支持 |
| **V3 完整版** | 根据验证结果决定是否恢复多 Agent 架构，替换为中文嵌入模型 |

## 14. Agent Pool：双面 Agent 与信息不对称调度

> 核心理念：每个 Agent 状态分为**公开面**（其他角色可通过观察获取）和**隐藏面**（内心世界，不可直接访问）。角色之间通过观察行为进行**推演**来猜测对方的隐藏状态，推演可能正确也可能错误——而戏剧冲突正诞生于这种信息差之中。

### 14.1 核心机制：双面 Agent 卡片

每个 Agent 的数据不再是一张平面的状态表，而是一张有正反面的卡片：

```
┌──────────────────────────┐     ┌──────────────────────────┐
│  🧑 林逸风 (公开面)       │     │  🔒 林逸风 (隐藏面)       │
│  ─────────────────────── │     │  ─────────────────────── │
│  外观: 面色苍白，话很少   │     │  真实HP: 25/100          │
│  公开情绪: 沉默           │     │  真实情绪: 恐惧 + 愧疚   │
│  位置: 清风镇客栈         │     │  隐藏目标: 保护师妹      │
│  已知关系:               │     │  秘密: 父亲失踪那晚      │
│    师妹: 最近很冷淡       │     │        他也在场          │
│  最近言行:               │     │  真实感情:              │
│    "我没事"              │     │    对师妹好感 70          │
│                          │     │    但刻意表现为疏远       │
│                          │     │                          │
│   ← 所有 Agent 可见 →    │     │   ← 仅作者/自己可见 →    │
└──────────────────────────┘     └──────────────────────────┘
```

#### 14.1.1 公开面字段定义

公开面是所有其他角色无需推理即可直接感知的信息：

| 字段 | 类型 | 说明 | 更新方式 |
|------|------|------|---------|
| 外观描述 | 文本 | 面色、体态、衣着等可见特征 | 状态变化时作者确认更新 |
| 公开情绪 | 枚举 | 外显的情绪表现（可能与隐藏面不同） | 情绪变化时自动生成 |
| 位置 | 地点引用 | 当前所在地点 | 移动时更新 |
| 最近的公开言行 | 对话/动作摘要 | 最近几轮对话中角色说了什么做了什么 | 每轮对话后自动追加 |
| 已知人际关系 | 数值 | 其他角色可感知到的态度（冷淡/友好/敌视） | 好感度变化时派生 |
| 所属势力/阵营 | 标签 | 公开的身份归属 | 手动设置 |
| 装备外显 | 列表 | 可见的武器、服装等 | 更换时更新 |

#### 14.1.2 隐藏面字段定义

隐藏面仅作者和 Agent 自己可访问：

| 字段 | 类型 | 说明 |
|------|------|------|
| 真实 HP / 内力 | 数值 | 实际生命值和能量值 |
| 真实情绪 | 枚举 + 强度 | 内心的真实感受 |
| 个人目标 | 文本 | 角色当前真正想达成的事情 |
| 秘密 | 文本列表 | 角色不为人知的信息 |
| 真实好感度 | 数值 | 对每个角色的真实感情（可能与公开表现不同） |
| 弱点/恐惧 | 文本列表 | 角色不愿被他人知道的弱点 |

#### 14.1.3 公开面与隐藏面的联动

```
     作者设置
        │
        ▼
   ┌───────────────────┐
   │     隐藏面 (真相)   │
   │   好感度 70       │  ← 真实数据，驱动 Agent 内心决策
   └───────┬───────────┘
           │
           │  行为策略: "刻意疏远"
           │
           ▼
   ┌───────────────────┐
   │     公开面 (表现)   │
   │   好感度表现为 "冷淡"│  ← 其他 Agent 只能看到这个
   └───────┬───────────┘
           │
           │  观察 + 推演
           │
           ▼
   ┌───────────────────┐
   │  Agent B 的认知模型 │
   │  "他讨厌我" 60%    │  ← 可能正确也可能错误
   └───────────────────┘
```

三层数据对应三种戏剧状态：
- **隐藏面 70 分** = 事实：他爱她
- **公开面表现为冷淡** = 表现：他看起来冷淡
- **对方解读为讨厌** = 误判：她以为他讨厌她

**冲突就诞生在这三层之间的差异里。**

### 14.2 认知模型：Agent 之间的互推演系统

每个 Agent 对其他每个 Agent 维护一个**认知模型 (Cognitive Model)**，包含观察记录、推理结果和确信度。这个模型可能正确，也可能错误。

#### 14.2.1 师妹眼中的"林逸风"

```
┌────────────────────────────────────────────┐
│  👤 师妹 对 林逸风 的认知模型                │
│  ──────────────────────────────────────────│
│  ▸ 直接观察 (来自公开面)                     │
│    外观: 面色苍白，话很少                    │
│    最近言行: "我没事"                       │
│    ──────────────────────────────          │
│  ▸ 推理结果                                 │
│    "他在隐瞒伤势"          确信度 85% ⬆     │
│    "和父亲失踪有关"        确信度 40%        │
│    "他在保护我"            确信度 25%        │
│    ──────────────────────────────          │
│  ▸ 可能为错误认知                            │
│    "他讨厌我"              确信度 60% ⚠     │
│    ──────────────────────────────          │
│  ▸ 待验证假设                               │
│    "他在客栈偷偷见了什么人"                   │
│    "他准备单独去找反派"                      │
└────────────────────────────────────────────┘
```

#### 14.2.2 推演引擎的运作流程

每当 Agent A 观察到 Agent B 的公开言行时：

```
Agent A 观察到: 林逸风说"我没事" + 面色苍白(来自公开面)
         │
         ▼
┌─────────────────────────────────────────────┐
│              推演引擎 (Inference Engine)       │
│                                              │
│  1. 对比公开面 vs 观察到的言行                 │
│     → "面色苍白" + "口头说没事" = 矛盾信号   │
│                                              │
│  2. 检索 Agent A 自己的相关记忆               │
│     → "上次他受伤也这样逞强" (模式匹配)       │
│                                              │
│  3. 结合 Agent A 的隐藏面状态                  │
│     → 师妹情绪: "担心" → 推演倾向偏保护性     │
│                                              │
│  4. 更新认知模型:                              │
│     → "他在隐瞒伤势" 确信度 +15% → 85%        │
│     → 新建假设: "林逸风当前 HP < 50"          │
│                                              │
│  5. 判断是否需要行为响应:                      │
│     → 确信度 > 70% 且为负面解读 → 可能触发    │
│       · 直接质问 (言语冲突)                   │
│       · 暗中观察 (等待更多信息)               │
│       · 向老铁匠打听 (第三方求证)             │
│       · 暂时不提 (但内心记下，影响后续行为)    │
└─────────────────────────────────────────────┘
```

#### 14.2.3 确信度的动态变化

| 事件 | 确信度变化 | 说明 |
|------|-----------|------|
| 观察到支持推论的言行 | +10%~+25% | 观察到的公开行为与推理一致 |
| 观察到矛盾的言行 | -15%~-30% | 观察到的公开行为与推理矛盾 |
| 获知隐藏面信息（通过对话/第三方） | +40%~+60% | 角色主动说出真相，或从第三方得知 |
| 时间流逝（未获得新信息） | -5%/章 | 随时间推移信念轻微衰减 |
| 强烈情绪事件 | ±20%~±40% | 在激烈冲突中，信念可能剧烈摇摆 |

### 14.3 作者视角：三种观察模式

作者在 Agent Pool 中可以切换三种视角来理解故事的信息结构：

| 视角 | 可见内容 | 用途 |
|------|---------|------|
| **上帝视角** | 所有 Agent 的公开面 + 隐藏面 + 所有认知模型 | 全面掌控叙事的全知叙事 |
| **角色 X 视角** | X 的隐藏面 + 所有其他人的公开面 + X 对所有人的认知模型 | 代入角色 X，理解 TA 为什么会那样行动 |
| **棋盘视角** | 仅公开面 + 认知差异高亮标注 | 发现"误会"和"信息差"——这就是故事冲突的矿脉 |

#### 棋盘视角示例

```
▎棋盘视角 — 当前信息差

  林逸风                         师妹
  ┌──────────────┐              ┌──────────────┐
  │ 🔒好感度 70   │              │ 🔒好感度 45   │
  │ 🔒HP 25      │              │ 🔒担心他     │
  │ 公开: 沉默    │              │ 公开: 冷淡    │
  └──────┬───────┘              └──────┬───────┘
         │                              │
         │  ⚠ 信息差:                  │
         ├──── 师妹误判 "他讨厌我" ────▶│
         │    确信度 60% ← 与事实矛盾!   │
         │                              │
         │◀── 林逸风没察觉              │
         │    师妹在担心他               │
         │                              │
   💡 推荐事件: "师妹忍不住质问: 你为什么躲着我?"
```

作者看到红色高亮的信息差，直接点击"推荐事件"按钮就能把这条认知差异变成一个调度事件。

#### 14.4.1 认知模型 JSON

  "目标": "hero_01",
  "推论列表": [

#### 14.4.2 推演触发策略

推演引擎并非每轮都全量运行，而是按需触发：

| 触发策略 | 触发时机 | 开销 |
|---------|---------|------|
| **事件驱动**（默认） | 观察到目标 Agent 的公开言行发生显著变化 | 低 |
| **定期运行** | 每章结束后，对所有活跃 Agent 关系做一次全量推演 | 中 |
| **作者手动触发** | 作者选中某个 Agent，右键 → "重新推演目标" | 低 |
| **实时预览** | 作者在棋盘视角中悬停在某个推论上 | 零 |

### 14.5 信息不对称驱动的调度

这是 Agent Pool 调度的核心升级——事件推荐不再只基于数值条件（好感度低于 30），而是基于**信息差**：

#### 14.5.1 信息差驱动的冲突类型

| 信息差类型 | 冲突模板 | 示例 |
|-----------|---------|------|
| **A 误解 B** | 质问型言语冲突 | 师妹认为林逸风讨厌她 → 质问 |
| **A 不知道 B 的秘密** | 揭露型场景 | 师妹偷听到林逸风和神秘人的对话 |
| **A 和 B 对同一事件的记忆不同** | 对质型冲突 | "那天晚上你在哪？"——两个版本的真相碰撞 |
| **A 不知道自己的认知是错的** | 戏剧反转型事件 | 林逸风最后说出真相："我不是讨厌你，我是怕连累你" |
| **C 知道 A 和 B 都不知道的事** | 第三方介入 | 老铁匠知道林逸风父亲失踪的真相 |

#### 14.5.2 调度面板升级

选中两个角色后，除了原有的"条件匹配事件"，新增"信息差事件"区域：

```
选中角色: [林逸风 ✓] [师妹 ✓]

▸ 信息差事件 (基于认知差异):
  ⚡ "你为什么躲着我？" — 师妹误判确信度 60%，林逸风隐藏好感度 70
     反转条件: 林逸风公开展示关心 或 师妹发现真相
  ⚡ "有人跟踪我们" — 林逸风知道这个信息，师妹不知道
  ⚡ 对质真相 — 两人对"父亲失踪那晚"的记忆不一致 (来自记忆对比)

▸ 状态匹配事件 (3):
  ● 激烈争吵 — 条件全部满足 ✓
  ● 冷战 — 条件全部满足 ✓
  ...
```

### 14.6 记忆的公开/隐藏属性

延续第 6 章的记忆系统，新增记忆的可见性标记：

| 记忆类型 | 谁可见 | 示例 |
|---------|-------|------|
| **公开记忆** | 所有共享该场景的 Agent | "林逸风在酒馆里摔了杯子" |
| **角色私有记忆** | 仅该 Agent 自己和作者 | "林逸风记得父亲走之前说：月圆之夜，别相信任何人" |
| **共享但表述不同** | 各 Agent 各有一份 | 林逸风版："是我先出手的" / 师妹版："林逸风先动手" |

私有记忆可以被"泄露"——在对话中角色主动说出后，被泄露的记忆自动复制到对方的认知模型中。

### 14.7 编辑机制：双面编辑

作者编辑 Agent 卡片时，可以分别编辑公开面和隐藏面：

```
作者选中 [林逸风]
    │
    ├── [编辑公开面] → 设置其他角色能看到什么
    │     - 外观描述
    │     - 公开情绪
    │     - 公开言行（手动添加/修改）
    │
    ├── [编辑隐藏面] → 设置角色的真实内心
    │     - 真实数值 (HP/内力/好感度)
    │     - 真实情绪
    │     - 秘密 / 目标 / 弱点
    │
    └── [行为策略] → 控制公开面与隐藏面的差异
          - 策略: [坦诚] [刻意疏远] [伪装友善] [隐瞒情绪]
          - 策略决定公开面如何从隐藏面派生
```

### 14.8 推演引擎的技术实现

#### 14.8.1 触发流程

```
事件: 林逸风公开言行更新
      │
      ▼
Agent B 的 Event Bus 收到通知: "林逸风公开言行已更新"
      │
      ▼
【规则层】先规则判断是否需要推演:
  - 公开情绪剧烈变化 (>30%偏移) → 触发
  - 公开言行与现有推论矛盾 → 触发
  - 无明显变化 → 跳过
      │
      ▼ (触发)
【LLM 推演】组装 Prompt → 轻量调用:
  Prompt 包含:
  - Agent B 的性格和当前情绪
  - B 对林逸风的现有认知模型
  - 最新公开观察数据
  - B 的相关记忆
  → 输出: 推论更新列表 + 确信度调整 + 行为响应建议
      │
      ▼
更新 B 的认知模型 → 通知前端
```

#### 14.8.2 成本控制

推演引擎的 LLM 调用成本远低于场景生成：

| 指标 | 场景生成 | 认知推演 |
|------|---------|---------|
| Prompt 长度 | 2000-4000 tokens | 500-800 tokens |
| 需要角色生成 | 长篇对话/动作 | 仅结构化判断 |
| 模型要求 | 高（需要文学质量） | 中（逻辑判断为主） |
| 可批量执行 | 否（逐个场景） | 是（每章结束批量更新所有认知模型） |
| 可用本地模型 | 勉强 | 完全可以 |

建议认知推演使用本地轻量模型，无需调用云端 API。

### 14.10 与被动模式的对比

| 维度 | 被动模式（原第8章） | Agent Pool (v3.1 初始版) | Agent Pool (双面卡版本) |
|------|-------------------|------------------------|----------------------|
| 冲突触发 | 系统检测 → 推荐 | 作者手动选择 | 作者选择 + 信息差高亮 |
| 角色状态 | 分散在各模块 | 统一面板 | 双面卡片（公开/隐藏） |
| 角色间互动 | 仅通过生成内容 | 仅通过生成内容 | 生成 + 认知推演 |
| 信息差利用 | 无 | 无 | 核心驱动机制 |
| 误解和反转 | 作者手动写 | 作者手动写 | 系统自动高亮认知差异 |
| 作者体验 | 编辑 + 审查 | 导演 + 操控 | 全知叙事者 + 任意角色视角 |
| 适合作者 | 大纲型 | 掌控型 | 追求戏剧张力和曲折剧情的作者 |

三种模式可以共存——左侧是主动调度（含信息差高亮），右侧是系统被动推荐通知栏。

### C. 冲突检测器规则示例（伪代码）

## 结束语

本系统通过 **状态追踪 + 时间轴 + 多角色独立记忆Agent + 环境Agent + 冲突驱动事件引擎**，实现了作者与AI的深度、有序配合。作者始终掌握创作主导权，AI则作为多角色副驾驶，帮助维持逻辑一致、激发冲突、丰富描写。

V3.2 的 Agent Pool 升级引入了双面 Agent 卡片和 Agent 互推演引擎，将**信息不对称**作为故事冲突的核心驱动机制——角色之间无法直接读取对方的内心，只能通过观察行为进行推演，而推演可能正确也可能错误。这种"隐藏面 vs 公开表现 vs 对方误读"的三层结构天然地产生戏剧张力和情节反转。核心建议不变：**先做 V1.0/V1.5（零 AI 依赖，风险为零），再做单 Agent 场景生成验证，最后根据验证结果决定架构方向。**
