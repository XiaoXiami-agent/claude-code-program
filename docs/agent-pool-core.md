# Agent Pool + Scene 核心设计

> 版本：1.0  
> 最后更新：2026-05-05  
> 定位：整个小说创作系统的 **MVP 最小单元**，独立于其他一切功能

---

## 1. 核心理念

砍掉一切，只做三件事：

| 概念 | 职责 | 比喻 |
|------|------|------|
| **Agent** | 角色的数字分身。有性格、有状态、有记忆，能生成符合人设的对话/动作 | 演员 |
| **Scene** | 共享的时空容器。定义"在哪、什么时间、什么气氛"，所有 Agent 可见 | 舞台 |
| **Pool** | 管理 Agent 生命周期，调度 Agent 之间的回合制交互 | 导演组 |

交互模式：作者选择若干 Agent → 放入 Scene → 选择交互类型 → 系统按回合制逐一调度 Agent 生成内容 → 作者确认。

**不做的**：自动冲突检测、向量记忆、时间轴、章节管理、Electron、导出。这些全部留给后续版本。

## 2. 数据模型

### 2.1 Agent

```typescript
interface Agent {
  id: string
  name: string
  personality: string        // 性格描述
  speakingStyle: string      // 语言风格

  state: {
    hp: number
    mp: number
    emotion: string
    location: string
    relationships: Record<string, number>   // { [agentName]: 好感度 }
    inventory: string[]
    buffs: string[]                         // ["左臂轻伤（持续2天）"]
  }

  shortMemory: MemoryItem[]  // 最近 20~30 条
  goals: string[]            // 当前目标，驱动行为
}

interface MemoryItem {
  id: string
  timestamp: string          // 故事时间
  content: string            // "师妹在暴雨夜指责我什么都不说，我很难过"
  tags: string[]             // ["冲突", "情感", "师妹"]
}
```

### 2.2 Scene

```typescript
interface Scene {
  id: string
  name: string               // 场景名称（作者自定义）
  location: string           // "清风镇客栈"
  time: string               // "戌时"
  weather: string            // "暴雨"
  atmosphere: string         // "压抑、紧张"
  background: string         // 场景背景描述，注入给所有 Agent

  participants: AgentId[]    // 当前场景中的 Agent
  interactionType: InteractionType
  maxRounds: number          // 最大回合数（停止条件 4）
  failureConditions: {       // 停止条件 2：失败阈值，可选
    hpThreshold?: number     // HP ≤ 此值视为失败（武力冲突默认 0）
    emotionExtreme?: string  // 情绪达到此值视为崩溃（如"崩溃""绝望"）
    affinityThreshold?: number // 好感度 ≤ 此值视为关系破裂
  }
  currentRound: number       // 当前回合
  status: 'idle' | 'running' | 'paused' | 'finished'
}
```

### 2.3 交互类型

```typescript
enum InteractionType {
  DIALOGUE = 'dialogue',         // 言语交互（默认）
  COMBAT = 'combat',             // 武力冲突
  MONOLOGUE = 'monologue',       // 内心独白（单 Agent）
  FREE_INTERACTION = 'free'      // 自由交互（不限定类型，Agent 自主决定行为）
}
```

│                    │                                 │
│              ┌─────▼──────┐                          │
│              │  Scheduler │  回合制调度引擎           │
│              └─────┬──────┘                          │
│                    │                                 │
├────────────────────┼─────────────────────────────────┤
│              ┌─────▼──────┐                          │

```

**关键技术决策**：
- 不使用多进程/多线程。Agent 就是 JS 对象实例。调度是单线程异步循环。
- Agent 不直接通信。所有消息经调度器中转，调度器决定"下一个该谁说话"。
- 状态修改不直接生效。Agent 输出"状态变化建议"，调度器解析后更新**临时状态**，作者确认后才写入正式状态。

## 4. 调度器（Scheduler）设计

调度器是 Agent Pool 的心脏。

### 4.1 回合循环

```
作者触发"开始交互"
       │
       ▼
┌─────────────────────────────────────────┐
│  1. 初始化 SimulationContext             │
│     - 深拷贝所有参与 Agent 的状态 → 临时状态 │
│     - 轮次计数器 = 0                     │
│     - 确定发言顺序                       │
├─────────────────────────────────────────┤
│  2. 回合循环:                            │
│     ┌─────────────────────────────────┐ │
│     │ a. 确定当前发言 Agent            │ │
│     │ b. 组装 Prompt（见第5节）        │ │
│     │ c. 调用 LLM，获取输出            │ │
│     │ d. 解析状态变化（见第6节）        │ │
│     │ e. 更新临时状态                   │ │
│     │ f. 推送给前端展示                 │ │
│     │ g. 检查四种停止条件（见 4.2）        │ │
│     │ h. 轮次 +1                       │ │
│     └─────────────────────────────────┘ │
├─────────────────────────────────────────┤
│  3. 达到停止条件 → 输出结果               │
│     - 完整对话文本                       │
│     - 状态变化汇总 (delta JSON)           │
│     - 等待作者确认                       │
└─────────────────────────────────────────┘
```

### 4.2 停止条件（四种，优先级从高到低）

每个回合结束后，调度器按以下优先级检查：

#### 1. 手动终止（优先级最高）

作者随时点击"停止"按钮。无论当前回合是否完成，立即中断 LLM 调用并冻结临时状态。这是作者的最高控制权——任何时候觉得方向不对、质量不满意，或只是想换个思路，都能一键停止。

#### 2. 失败（状态归零）

任意参与 Agent 的某项关键数值触底：

| 交互类型 | 失败条件 |
|----------|---------|
| 言语交互 | 情绪达到"崩溃"（emotion 判定为极值）、好感度降至 0 以下 |
| 武力冲突 | HP ≤ 0、失去战斗能力 |
| 内心独白 | 无失败条件（独白自然结束或手动终止） |
| 自由交互 | 由交互类型动态决定 |

触发失败时，调度器标记该 Agent 为"落败"，不再给败者分配发言回合。剩余 Agent 可以继续（如胜者说最后一句话），也可以立即结束。

#### 3. 避让退出（Agent 自主选择离开）

Agent 在生成内容时可以输出**退出意图**，表示该角色主动选择结束交互：

- 言语交互中：角色说出类似"我不想再说了""今天就到这里"的对话 → 二次解析时识别到退出意图
- 武力冲突中：角色"跳出战圈""转身离去""加速逃离"
- 自由交互中：角色"离开房间""借故离去"

实现方式：状态解析阶段（见第 6 节）额外返回一个 `exit_intent: boolean` 字段。Agent 本身不需要显式输出 JSON——它只管写对白，对白的内容自然包含了退出信号，由二次解析来识别。

避让退出不等同于失败——退出的角色可能只是"不想吵了"而非输了。退出的 Agent 从当前回合列表中移除，剩余 Agent 可以继续或自行结束。

#### 4. 限制会话交互次数（回合上限）

达到 Scene.maxRounds。这是一个保底机制，防止无限循环或 API 消耗失控。回合上限是作者配置的参数，可以在交互开始前调整（默认值建议 6-10 轮）。

回合上限触发后，调度器给当前发言人最后一个回合做收尾，然后强制结束。

---

**四种条件的关系**：
- 手动终止可以覆盖一切（优先级最高）
- 失败和避让退出是 Agent 驱动的自动结束
- 回合上限是系统兜底
- 条件 2/3/4 任意一个触发，都会进入"结束流程"：输出对话文本 + 状态变化汇总 + 等待作者确认

### 4.3 发言顺序

| 交互类型 | 顺序规则 |
|----------|---------|
| 言语交互 | 由"当前情绪强度高"的一方先开口，然后轮流 |
| 武力冲突 | 按"速度/敏捷"排序（可在 Agent 状态中加 speed 字段） |
| 内心独白 | 单个 Agent 与环境 Agent 交替（独白 → 环境渲染 → 独白） |
| 自由交互 | 调度器自行判断"谁更可能主动行动"，或随机 |

  情绪：{agent.state.emotion}

【当前场景】
  地点：{scene.location}
  时间：{scene.time}
  天气：{scene.weather}
  氛围：{scene.atmosphere}
  场景描述：{scene.background}

【场景中的其他角色】
{其他参与者列表，含姓名、性格简介、与你的好感度}

【你的近期记忆】（最近 5 条）
{agent.shortMemory 的最近 5 条}

```

## 6. 状态变化解析

Agent 输出的是自然语言（对话 + 动作）。需要额外一步将自然语言转为状态变化数值。

### 方案：二次调用轻量模型

Agent 的原始输出：
> （扭过头去，声音发抖）你每次都这样，什么都不肯说！我不想再听了。

将此输出 + 当前状态发给一个**轻量 LLM 调用**（或同一个 LLM 的简短调用），要求输出结构化结果：

```
分析以下角色对话，提取状态变化（仅输出 JSON，不要解释）：

【当前状态】
情绪：愤怒
好感度：当前 75

【对话内容】
"（扭过头去，声音发抖）你每次都这样，什么都不肯说！我不想再听了。"

---
{
  "emotion": "混合：表面愤怒，深层悲伤",
  "relationship_change": -8,
  "hp_change": 0,
  "exit_intent": false,
  "reason": "表达长期压抑的失望"
}
```

若对话中出现明显的退出信号（"我不想再说了""今天就到这里"、摔门离开、转身离去等），`exit_intent` 设为 `true`，触发停止条件 3（避让退出）。

### 为什么这样做

- Agent 专注于"写对白"，不操心 JSON，质量更高
- JSON 解析失败的容错成本低（重试一次即可）
- 轻量调用比角色 Agent 调用快得多（短 Prompt，短输出）
- 如果作者实时看着，也可以用本地小模型做解析，不额外消耗云端额度

### 交互要点

- **Agent 卡片可拖拽**：拖到 Scene 区域即添加为参与者
- **单步模式**：每回合暂停，作者审核每一句对话，可逐句修改
- **自动模式**：连续生成，作者只看最终结果
- **暂停干预**：在任何回合暂停，手动调整某个 Agent 的状态数值，然后继续
- **状态编辑器**：点击 Agent 卡片上任意数值，弹出编辑框直接修改

## 8. 存储方案

MVP 阶段全部用 JSON 文件：

```
project/
├── agents/
│   ├── 林逸风.json
│   ├── 师妹.json
│   └── 反派.json
├── scenes/
│   ├── 客栈争吵.json
│   └── ...
├── pool-config.json          // Pool 全局配置
└── snapshots/
    └── 2026-05-05_1500/      // 每次确认后的状态快照
        ├── 林逸风.json
        └── 师妹.json
```

### 读写策略

- Agent 对象在**内存**中，启动时从 JSON 加载
- **每次交互确认后**写入 JSON（存档）
- 交互过程中只操作内存中的**临时状态**，不写盘
- 浏览器环境用 IndexedDB 替代文件系统

## 9. 技术栈（极简版）

| 层 | 选择 | 理由 |
|----|------|------|
| 运行时 | Node.js（CLI 原型）或浏览器单页 | 不需要框架也能跑 |
| Agent 实例 | JS Class | 状态 + 方法，直观 |
| 调度器 | async generator | `for await (const turn of scheduler.run(scene, agents))` |
| LLM API | OpenAI SDK / Anthropic SDK | 用户自备 Key |
| 存储 | 文件 JSON 或 IndexedDB | 零依赖 |
| 前端（可选） | React + Zustand | 轻量状态管理 |

**原型可以在一个 HTML 文件里跑起来**。不需要 Electron、不需要数据库、不需要向量库。

## 10. MVP 实现步骤

| 步骤 | 内容 | 产出 |
|------|------|------|
| **S1 数据结构** | 定义 Agent、Scene、MemoryItem 的 interface/class | 可运行的 JS 类 |
| **S2 单 Agent 对话** | 给 Agent 注入 Prompt，调 LLM，获取一句话 | 命令行看到 Agent 输出 |
| **S3 回合循环** | 两个 Agent 轮流说话，3 轮后停止 | 命令行看到完整对话 |
| **S4 状态解析** | 每次对话后解析好感度/情绪变化 | 命令行看到数值变化 |
| **S5 状态编辑** | 作者可在回合间隙修改 Agent 状态 | 可中断修改 |
| **S6 记忆队列** | 交互确认后写入短时记忆 | Agent 记住发生过的事 |
| **S7 简单 UI** | React 页面：Agent 卡片 + Scene 面板 + 对话流 | 可视化操作 |
| **S8 存档恢复** | JSON 读写，支持"加载上次进度" | 数据持久化 |

## 11. 与完整版系统的关系

本核心模块是 `novelsystem-v3.md` 中：

- 第 5 章（多角色 Agent 系统）的 **极简实现**
- 第 8 章（冲突引擎）中事件编排器的 **主动触发版本**
- 第 14 章（Agent Pool）的 **落地执行方案**

完整版的其他模块（时间轴、自动冲突检测、向量记忆、环境 Agent 风格预设等）可以作为 Scene 的**增强插件**逐步叠加，但核心引擎不再改动。

## 12. 风险与边界

| 风险 | 应对 |
|------|------|
| LLM 输出偏离人设 | Prompt 收紧 + 给"反例"（你不该说的话） |
| 状态解析数值不准 | 解析失败的回合显示"？"，作者手动填写 |
| 长对话 Agent 失忆 | 短时记忆保留最近 5 条对话，Prompt 中始终可见 |
| 三个以上 Agent 调度混乱 | MVP 先限制 2-3 个参与者，验证后再扩展 |
| 云端 API 调用慢 | 单步模式让作者边看边等，体验上不显得慢 |
